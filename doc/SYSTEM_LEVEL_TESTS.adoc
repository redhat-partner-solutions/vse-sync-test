= System-Level Tests

This document describes the system-level synchronization tests in this repository, including their purpose, duration requirements, and how they differ from functional tests.

== Overview

There are two categories of "system-level" tests in this repository:

* *Path validation tests* (`system-test-PHC-to-SYS`): Validate the complete end-to-end synchronization path from PHC to System Clock (SYS). They apply the full 100% requirement at the final observation point but do *not* run a workload.

* *Workload-based system tests* (`ptp_workload_system_test`): A workload that *consumes PTP events*, *stresses the system*, and *validates clock accuracy* under load. These simulate RAN-style consumers that continuously read PTP timestamps while the system is under stress, ensuring the clock meets requirements (e.g. < 100 ns for RAN) during real-world conditions.

The path validation tests verify the sync chain in isolation. The workload-based tests verify that the clock remains accurate when the system is actively consuming PTP and under load.

== Test Duration

=== Summary

| Parameter | Value | Description |
|-----------|-------|-------------|
| Transient period | 300 seconds | Initial samples ignored (lock acquisition) |
| Minimum test duration | 1000 seconds | Minimum valid test window |
| Recommended data collection | 2000 seconds | Default E2E collection duration |
| Total minimum run time | ~1300 seconds | Transient + minimum test window |

=== Detailed Duration Parameters

All system-level tests use the following configuration parameters (defined in `config.yaml`):

* *transient-period/s: 300* — Samples in the first 300 seconds are discarded to allow the system to reach steady-state locked conditions.
* *min-test-duration/s: 1000* — The test window (after transient) must be at least 1000 seconds for a valid result.

=== E2E Pipeline

When running the full end-to-end pipeline (`cmd/e2e.sh`):

* *Default collection duration: 2000 seconds* — Data is collected for 2000 seconds by default.
* *Configurable* — Use the `-d` option to override: `./e2e.sh -d 3600s ~/kubeconfig`

For TDEV and MTIE tests, longer collection periods may improve coverage of higher tau (observation window) values. Per ITU-T G.811, the minimum test window for TDEV is twelve times the maximum observation interval; the implementation uses a 1000-second minimum for practical test execution.

== Test Naming Convention

All system-level tests use the prefix `system-test-PHC-to-SYS` to distinguish them from functional tests. The test identifier format is:

```
sync/<standard>/<metric>/system-test-PHC-to-SYS/<variant>
```

Example: `sync/G.8272/time-error-in-locked-mode/system-test-PHC-to-SYS/PRTC-A`

== Data Source

All system-level tests use `phc2sys` log data from the PTP daemon. The log captures the time error between the PHC and the system clock, which represents the final output of the synchronization chain.

* *Input file:* `linuxptp-daemon-container.log` (or equivalent PTP daemon log)
* *Parser:* phc2sys TimeErrorParser

== Functional vs System-Level Tests

| Aspect | Functional Tests | Path Validation (PHC-to-SYS) | Workload-Based System Tests |
|--------|------------------|------------------------------|-----------------------------|
| **Observation point** | Single segment (e.g., DPLL-to-PHC) | Full PHC-to-SYS path | PHC under workload |
| **Budget** | Partial (e.g., 10%) | Full 100% | RAN limit (100 ns) |
| **Purpose** | Isolate failures to specific segments | Validate end-to-end path | Validate clock under load |
| **Workload** | None | None | Consumes PTP, stresses system |
| **Data source** | Varies by segment | phc2sys logs | phc_ctl cmp during workload |
| **Location** | Segment name (e.g., DPLL-to-PHC) | system-test-PHC-to-SYS | ptp_workload_system_test/ |

== Workload-Based System Tests (ptp_workload_system_test)

The workload-based system tests run a workload that:

1. *Consumes PTP events* — Continuously reads PHC timestamps (`phc_ctl cmp`)
2. *Stresses the system* — Optional CPU load to simulate RAN workload
3. *Validates clock* — Ensures time error stays < 100 ns (RAN limit) during the workload

See `ptp_workload_system_test/README.md` for usage. Run via:

* *E2E pipeline* — Automatically runs when executing `./cmd/e2e.sh` with kubeconfig (e.g. `podman run ... ./vse-sync-test/cmd/e2e.sh /usr/vse/kubeconfig`). Results appear in the PDF report.
* `./run_workload_test.sh` — Standalone: executes inside linuxptp-daemon pod
* `oc apply -f ptp_workload_job.yaml` — Kubernetes Job (in-cluster)

== Running System-Level Tests

Path validation tests are included automatically when running the E2E pipeline:

* *T-GM mode:* `./e2e.sh -m gm ~/kubeconfig`
* *T-BC mode:* `./e2e.sh -m bc ~/kubeconfig`

To extend the collection duration for more robust TDEV/MTIE results:

```
./e2e.sh -d 3600s -m gm ~/kubeconfig
```

== Acceptance Criteria

*Path validation tests (PHC-to-SYS):*

1. All samples in the test window meet the full (100%) requirement for the metric.
2. The time output remains locked for the entire test window.
3. The test window is at least 1000 seconds long (after transient).

*Workload-based system tests:*

1. Max time error during workload < 100 ns (RAN limit).
2. Workload runs for configured duration (default 300 s) consuming PTP and stressing system.
